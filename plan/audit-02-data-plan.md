# Data Layer Redesign Plan ‚Äî F2L Case Database

Authored 2026-04-24. Addresses P0-1 through P0-4, STD-2, STD-3 from audit-01-results.md.

---

## 1. Root Cause Diagnosis

All five data P0 issues share the same origin: the JSON was generated by taking a known
algorithm and copying it verbatim into both `scramble` and `algorithms[0].moves`. This means
the viewer is asked to apply the algorithm as a setup move, rotate to a "starting" state that
is actually a random intermediate position, and then solve from there ‚Äî which makes no sense.

The schema design is sound. The data is wrong. No field needs to be added to fix the
fundamental viewer problem. What is needed is:

1. Correct setup values in every case.
2. A field rename to make the intent unambiguous so this mistake cannot happen again.
3. Structural fixes for impossible, duplicate, and mislabeled cases.

---

## 2. Schema Changes

### 2.1 Rename `scramble` ‚Üí `setup`

**Recommendation: rename the field.**

`scramble` is ambiguous ‚Äî it sounds like a competition scramble (random state). The field's
purpose is to set up a specific pedagogical starting state for the viewer. The word `setup`
conveys that intent directly. Every consumer of the JSON currently reads `case.scramble`; this
is a one-line change in `cases.js` with high clarity payoff.

Do not add a field called `inverse_alg`. That name implies the field is derived from the
algorithm, but setup and algorithm are not always strict mathematical inverses: for cases with
multiple viable setups (see section 4), the setup is chosen for pedagogical clarity, not
computed mechanically.

Do not store both the algorithm and its inverse as separate fields. The JS can compute an
inverse when needed (for display or copy-to-clipboard), and storing both creates a data
integrity trap where they can drift out of sync.

### 2.2 What `setup` must contain

`setup` must be the **inverse** of the primary algorithm, expressed in standard WCA notation,
relative to the **FR slot** with white on bottom. When applied to a solved cube, `setup`
produces exactly the starting position that `algorithms[0].moves` solves.

For a simple algorithm like `R U' R'`, the inverse is `R U R'`. For `U R U' R'`, the inverse
is `R U R' U'`. Inverses are computed by reversing the move order and inverting each move
(R‚ÜíR', U2‚ÜíU2, R'‚ÜíR).

### 2.3 Add `recognition` back as a top-level field

The current JSON uses `recognition` as a field name (present in the existing data, absent from
the phase-2 schema doc). Keep it. It is the right name and it belongs at the top level.

### 2.4 Remove `description` from the schema

Phase-3-data.md specifies a `description` field. The live JSON does not have it; `recognition`
covers the same ground more specifically. Do not add `description`. Use `recognition`.

### 2.5 Final schema (authoritative)

```jsonc
{
  "cases": [
    {
      "id": "f2l-01",           // string ‚Äî stable identifier, see section 6 on numbering
      "name": "...",            // short human-readable label, rendered in the card header
      "category": "easy",       // easy | corner-in-slot | corner-in-top | both-in-top | advanced
      "slot": "FR",             // always FR ‚Äî all cases are expressed relative to FR slot
      "setup": "R U R' U'",    // INVERSE of primary algorithm; sets up the case in the viewer
      "recognition": "...",     // how to identify this case mid-solve (plain text for now)
      "algorithms": [
        {
          "label": "Main",
          "moves": "U R U' R'",
          "notes": ""
        },
        {
          "label": "Alt",
          "moves": "...",
          "notes": "Mirror / fewer moves from certain AUF"
        }
      ]
    }
  ]
}
```

No other fields. Nothing is added that cannot be computed from these fields.

## 3. The AUF Variants (f2l-03 and f2l-04)

**Recommendation: remove f2l-03 and f2l-04 entirely.**

`U2 R U' R'` is `f2l-01` with `U2` prepended as alignment. `U2 F' U F` is `f2l-02` with `U2`
prepended. These are not distinct cases by any rigorous geometric definition: the corner-edge
pair is in the same relative orientation; only the U-layer AUF angle differs.

The SpeedSolving wiki, Brian Brewster's F2L guide, and Badmephisto's F2L tutorial all agree:
AUF before insertion is not a separate case. Including them inflates the count from 39 to 41,
misleads learners about how many cases they need to learn, and directly contradicts STD-3.

The site does benefit from acknowledging AUF conceptually ‚Äî but through a note on the `easy`
category, not by listing variants as separate entries. Add a `notes` field at the category
level (in a separate `categories` key in the JSON, or in the about.html prose) that reads:
"For paired cases, AUF before insertion is not a new case. Simply turn the U layer to align
the pair above the correct slot, then apply the insertion."

After removal, the `easy` category has 2 entries (f2l-01 and f2l-02). That is correct.

## 4. Same Algorithm, Multiple Cases (P0-4)

The three cases f2l-12, f2l-22, and f2l-33 all genuinely use `R U' R' U R U' R'`. Likewise
f2l-13 and f2l-34 both use `F' U F U' F' U F`. These are not copy-paste errors.

**Explanation:** In F2L, the same trigger sequence can reach a solved slot from different
starting configurations when the corner and edge happen to land in the same intermediate
position mid-algorithm. The algorithm is doing different "work" in each case ‚Äî it's just that
the position after all setup moves is coincidentally the same.

**Recommendation: keep distinct cases, add a `shared_algorithm_note` field.**

These cases should remain as separate entries because their **starting states are different**
and a learner needs to recognize them differently. The fact that one algorithm solves all three
is itself a useful pedagogical observation ‚Äî it reduces memorization load.

Add an optional field `shared_algorithm_note` (string, omitted when not applicable):

```jsonc
{
  "id": "f2l-22",
  "shared_algorithm_note": "Same algorithm as f2l-12 and f2l-33. Different starting positions converge to the same intermediate state."
}
```

This makes the relationship explicit in the data so the UI can surface it (e.g., a small badge
reading "Algorithm shared with 2 other cases") without hard-coding that logic in JS.

## 5. Fixing f2l-41 (Impossible Case)

**Recommendation: replace f2l-41 with the correct advanced case it was meant to represent.**

The current f2l-41 describes "corner in slot flipped, edge in slot flipped" and provides
`(R U R' U') √ó 3` as the algorithm. Both parts of this are wrong:

- The case is geometrically impossible. A middle-layer edge cannot be flipped in place without
  disrupting other pieces. This state cannot exist in a valid 3√ó3 solve.
- `(R U R' U') √ó 3` is a well-known OLL orientation trick, not an F2L algorithm.

The "advanced" category has 3 remaining real cases (f2l-38, f2l-39, f2l-40) after removal of
f2l-41. Rather than invent a replacement, reduce the advanced category to 3 entries. If a
genuine 4th advanced case is wanted, the correct candidate is: **corner in FR slot
misoriented, edge in an adjacent wrong slot (BR or FL)**. That case exists, is geometrically
valid, and requires an 11-move extraction. It can be added once a qualified cuber verifies
the algorithm.

For now: **delete f2l-41, document the deletion in a comment at the top of the JSON.**

## 6. Category Rename: `edge-in-top` ‚Üí `corner-in-top`

**Recommendation: rename the category immediately.**

Cases f2l-30 through f2l-37 (or their renumbered equivalents) all have the **corner in the
top layer** and the **edge already in a middle slot**. The current category name `edge-in-top`
is the exact opposite of what the cases contain. This is P1-4 in the audit.

New category key: `corner-in-top`  
New display label: "Corner in Top"  
New description: "Corner is free in the U layer; edge is already in a middle slot."

Update: the JSON `category` values, the filter button labels in cases.html, the about.html
category descriptions, and any CSS selectors that key off the category string.

## 8. Numbering: Custom Sequential vs. SpeedSolving Wiki Alignment

**Recommendation: keep custom sequential numbering, fix the attribution.**

The SpeedSolving wiki uses a specific numbering convention (cases 1-41, derived from Lars
Vandenbergh's original classification). Aligning to that numbering would require:

- Adopting the wiki's exact case ordering, which uses a different primary sort axis
- Accepting gaps or non-sequential IDs if any cases are excluded
- Coupling this site's content decisions to an external resource that can be edited by anyone

The tradeoffs do not favor alignment. The wiki numbering is not pedagogically ordered ‚Äî
it is a reference index. This site is a teaching tool. The cases are deliberately ordered
from easy to hard, which is a different and valid organizing principle.

What must change is the **attribution**. STD-2 in the audit correctly flags the footer text:
"Algorithms sourced from the SpeedSolving wiki" is misleading when no IDs, ordering, or
category names match the wiki. Change the attribution to:

> "Algorithms drawn from standard CFOP practice. Case structure follows a pedagogical
> ordering independent of the SpeedSolving wiki's reference numbering."

The `id` field format `f2l-NN` should remain, with sequential numbering restarting from 01
after the removals are applied (so there are no gaps).

## 9. The `recognition` Field: Plain Text vs. Decision Tree

**Recommendation: keep plain text now, design for structured upgrade later.**

A decision-tree `recognition` structure (e.g., nested objects describing "find white sticker
‚Üí check face ‚Üí determine edge orientation") is the right long-term design (it addresses
PED-3 from the audit). It is also a significant content authoring effort that touches all 38
cases and requires a UI component to render the tree.

The immediate priority is getting the data geometrically correct (setup values, removed cases,
impossible case). Do not redesign the recognition format in the same pass.

When the recognition upgrade is tackled: the field should become an array of decision-step
objects, but the plain-text `recognition` string should be preserved alongside it for
fallback rendering. Schema-wise, a `recognition_steps` array (new field, optional) alongside
the existing `recognition` string is the safest migration path.

## 10. Implementation Order

Fix these in strict order. Each step produces a valid, shippable JSON file.

**Step 1 ‚Äî Rename `scramble` to `setup` in JSON and in `cases.js`.**  
Low risk. No content change. Immediately clarifies intent.

**Step 2 ‚Äî Correct all `setup` values to be inverses of their primary algorithm.**  
This is the bulk of the work. All 38 remaining cases need their setup string recomputed.
Cases where `algorithm == setup` are the clearest bugs; fix those first.
Verify each case by hand or with a cube simulator before committing.

**Step 3 ‚Äî Remove f2l-03 and f2l-04. Remove f2l-41.**  
Three deletions. Renumber remaining cases sequentially. Update progress.js denominator.

**Step 4 ‚Äî Rename category `edge-in-top` ‚Üí `corner-in-top` everywhere.**  
JSON, JS filter list, HTML display labels, about.html prose.

**Step 5 ‚Äî Add `shared_algorithm_note` to f2l-12/22/33 and f2l-13/34 equivalents.**  
Content-only addition. No schema change required; field is optional.

**Step 6 ‚Äî Fix the footer attribution text.**  
Single line change in HTML.

## 11. What This Plan Defers

These are explicitly out of scope for this data-layer fix:

- Structured `recognition_steps` decision tree (noted in section 9 for future phase)
- Adding a genuine 4th advanced case to replace f2l-41 (needs algorithm verification)
- Slot variants (FL, BR, BL) ‚Äî all cases remain FR-relative; mirroring is a UI concern
- AUF pre-notes as structured data ‚Äî prose in about.html is sufficient for now
- Any changes to algorithm content beyond correcting setup strings
